# 🚀 [Devlog] Hidden-Gem Finder: 단순 검색을 넘어선 AI 기반 게임 큐레이션 시스템 아키텍처 구축기

## 1. 프로젝트 개요 및 기획 의도

* **프로젝트명:** Hidden-Gem Finder (숨겨진 명작 발굴기)
* **기획 의도:** 기존 게임 플랫폼의 추천 시스템은 '단순 태그 매칭'이나 '인기 순위'에 편향되어 있습니다. 유저가 **"위쳐처럼 스토리가 깊고 판타지 세계관인 오픈월드 게임"**이라고 자연어로 검색했을 때, 문맥의 의미를 파악하여 대중성은 낮아도 작품성이 뛰어난 '숨겨진 보석'을 찾아주는 AI 추천 엔진을 설계했습니다.
* **Tech Stack:**
    * **Backend:** FastAPI, Python, SQLAlchemy
    * **Database:** PostgreSQL, pgvector (Vector DB)
    * **AI / ML:** OpenAI (text-embedding-3-small), Upstage Solar LLM, sentence-transformers
    * **Frontend:** Next.js, Tailwind CSS

---

## 2. 핵심 알고리즘 및 데이터 파이프라인 (Data Pipeline)

추천의 질을 높이기 위해 단순 크롤링을 넘어 **[수집 -> AI 가공 -> 벡터화 -> 하이브리드 검색]**의 4단계 파이프라인을 구축했습니다.

### # Step 1. 데이터 수집 (RAWG & Steam API)
* 초기 RAWG API로 1,000개의 게임을 수집했으나, 유저 평가 데이터의 깊이 부족을 인지하고 Steam API로 전환하여 공식 한국어 설명과 유저 추천 수(인기 척도)를 확보했습니다.

### # Step 2. LLM 기반 데이터 가공 (Data Enrichment)
* 단순히 설명만 저장하지 않고, Upstage Solar LLM(solar-mini)을 활용해 게임 설명 텍스트를 분석했습니다.
* 프롬프트 엔지니어링을 통해 `mania_score`(매니아 점수), `story_depth`(스토리 깊이) 등 12가지 독자적인 정량화 지표를 추출하여 메타데이터로 저장했습니다.

### # Step 3. 텍스트 임베딩 (Vectorization)
* 유저의 자연어 검색어와 게임의 상세 설명을 다차원 벡터 우주 공간의 좌표로 변환하여, 단어가 달라도 문맥이 비슷하면 매칭되도록 구성했습니다.

### # Step 4. 하이브리드 스코어링 (Hybrid Search)
* AI 유사도 점수에 LLM이 추출한 Hidden-Gem 메타데이터 가중치를 합산하여 최종 순위를 도출합니다.

## 3. 🚨 딥다이브 트러블슈팅 (Troubleshooting & Refactoring)

프로젝트를 진행하며 겪은 치명적인 한계점들과 이를 극복하기 위한 아키텍처 개선 과정입니다.

### # Issue 1: AI 라벨링 중 LLM의 할루시네이션(Hallucination)에 의한 파이프라인 중단
* **상황:** 500개 이상의 게임 데이터를 LLM에 통과시켜 정수(1~10) 점수를 추출하는 루프를 실행함.
* **문제:** "숫자만 출력하라"는 강력한 시스템 프롬프트에도 불구하고, 간헐적으로 "이 게임의 점수는 8점입니다"와 같이 문자열을 포함해 반환하여 파이썬 형변환 에러(`invalid literal for int()`)가 발생, 전체 배치 작업이 중단됨.
* **해결 (Shooting):**
    * 파이썬 정규표현식을 도입하여 반환값 방어 로직 구축. `re.findall(r'\d+', ai_reply)` 코드를 통해 응답 문장 내에서 연속된 숫자만 강제 추출함.
    * 추출 실패 시 Exception 처리와 함께 기본값(5점)을 부여하는 Fail-safe 로직을 추가하여 대규모 데이터 처리 안정성을 확보함.

### # Issue 2: 한국어 검색 시 의미론적 공간(Semantic Space) 붕괴 현상
* **상황:** 초기 임베딩 모델로 가볍고 빠른 오픈소스인 `sentence-transformers/all-MiniLM-L6-v2` (384차원)를 채택함.
* **문제:** 영어 텍스트 기반으로는 잘 작동했으나, 프론트엔드에서 한국어로 "액션" 검색 시 장르 연관성이 아예 없는 '두근두근 문예부(심리 공포)'가 1순위로 반환되는 치명적 오류 발생.
* **원인 분석:** 해당 로컬 모델은 영어 말뭉치로만 학습되어 한국어 "액션"을 노이즈로 인식, 벡터 공간에서 엉뚱한 좌표에 매핑해버림.
* **해결 (Shooting):**
    * 아키텍처 전면 개편 단행. 로컬 모델을 걷어내고 다국어 문맥 매칭이 압도적인 OpenAI `text-embedding-3-small` (1536차원) API로 두뇌를 교체.
    * DB 스키마 역시 384차원에서 1536차원 벡터를 수용하도록 PostgreSQL 테이블 재설계 진행. 결과적으로 "타격감 있는 액션" 같은 한국어 뉘앙스를 영어 게임 설명과 완벽하게 매칭시키는 데 성공함.

### # Issue 3: Min-Max 정규화의 함정과 점수 왜곡 (알고리즘 최적화)
* **상황:** Vector DB(pgvector)에서 코사인 거리(`<=>`)로 추출한 유사도 점수의 편차가 너무 작아, 기본 스탯(매니아 점수 등)이 높은 게임이 검색어와 무관하게 항상 상위권을 독식함.
* **해결 (Shooting):**
    * 쿼리 결과 내에서 유사도 점수의 격차를 강제로 벌려주는 동적 Min-Max Scaling 알고리즘을 도입.
    * 검색된 상위 30개 게임 중 가장 거리가 가까운 게임은 80점, 가장 먼 게임은 40점으로 스케일링(`normalized_ai_score`)을 적용.
    * 최종 점수를 **[AI 유사도 점수(80점 만점) + Hidden-Gem 보너스 가중치(20점 만점)]**로 튜닝하여, 검색어와의 연관성을 절대적으로 유지하면서도 숨겨진 명작이 약간의 어드밴티지를 받도록 하이브리드 로직을 완성함.

### # Issue 4: PostgreSQL Vector 타입 인식 불가 에러
* **상황:** SQLAlchemy를 통해 DB 초기화 스크립트 실행 중 `type "vector" does not exist` 에러 발생.
* **문제:** Docker 컨테이너 내에서 pgvector 확장이 설치는 되어 있으나, 특정 DB 인스턴스에서 활성화되지 않음.
* **해결 (Shooting):** 테이블 생성 쿼리 이전에 `CREATE EXTENSION IF NOT EXISTS vector;` 구문을 실행하도록 DB 초기화 코드를 수정하여 자동화 인프라의 안정성을 높임.

## 4. 프로젝트의 의의 (Learned & Impact)

단순히 API 문서를 읽고 호출하는 것을 넘어, **"데이터의 품질이 AI 알고리즘의 한계를 결정한다"**는 사실을 뼈저리게 배웠습니다. LLM의 불확실성을 정규표현식으로 통제하고, 다국어 처리의 한계를 모델 교체와 차원 확장으로 극복하며 '실제 유저에게 유의미한 결과를 내는 백엔드 서버'를 구축하는 아키텍처 설계 능력을 길렀습니다.
